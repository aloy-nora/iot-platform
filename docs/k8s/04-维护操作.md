# 04 · 维护操作

## 让新配置生效（改了清单后）

**改了 Deployment 本身**（`command`/`env`/`image` 等 pod 模板字段）→ **`apply` 一条就够**，它会自动滚动重建 Pod：
```bash
kubectl apply -f deploy/k8s/
kubectl rollout status deploy/redis -n iot   # 看滚动进度
```

**只改了 Secret/ConfigMap 的值、Deployment 模板没动** → `apply` 更新了 Secret，但运行中的 Pod 是启动时读的 env，不会自动刷新，需手动重建：
```bash
kubectl rollout restart deploy/redis -n iot
```

> `rollout restart` 的本质=模板没变时强制重建 Pod（典型场景：引用的 Secret/ConfigMap 值变了）。
> **模板变了则 `apply` 本身已触发滚动，不用再 restart**（否则是多余的第二次重启）。

## 更新镜像 / 回滚

```bash
kubectl set image deploy/backend backend=iot-backend:v2 -n iot   # 换镜像
kubectl rollout status deploy/backend -n iot                     # 等就绪
kubectl rollout undo deploy/backend -n iot                       # 出问题回滚
kubectl rollout history deploy/backend -n iot                    # 版本历史
```

## 扩缩容

```bash
kubectl scale deploy/backend --replicas=3 -n iot     # 无状态服务可扩
```
> ⚠️ MySQL/Redis 这类**有状态单实例别随便扩 replicas**（多副本抢同一 PVC 会坏数据）。真要多副本得用 StatefulSet + 各自 PVC。

## 持久化验证（确认数据不随 Pod 消失）

```bash
# 1. 写数据
kubectl exec -it -n iot deploy/mysql -- mysql -uroot -pexample123 iot \
  -e "CREATE TABLE t(id INT); INSERT INTO t VALUES(42);"
# 2. 删 Pod（Deployment 自动重建新 Pod）
kubectl delete pod -n iot -l app=mysql
kubectl get pods -n iot -w        # 等新 Pod 1/1
# 3. 数据还在 → PVC 生效
kubectl exec -it -n iot deploy/mysql -- mysql -uroot -pexample123 iot -e "SELECT * FROM t;"
```

## 删除与重建

```bash
kubectl delete pod -n iot -l app=redis        # 删 Pod（Deployment 会拉起新的）
kubectl delete -f deploy/k8s/20-redis.yaml     # 按清单删（含 Deployment/Service/PVC/Secret）
kubectl delete namespace iot                   # 清空整个命名空间（连 PVC 数据一起没）
```
> 注意：`delete -f` 若清单含 PVC，会**连数据一起删**。只想换 Pod 用 `delete pod` 或 `rollout restart`。

## 初始化表结构（schema）

MySQL/TDengine 的 `iot` 库建好后是**空的，表要手动灌**（MySQL 的 `MYSQL_DATABASE`、TDengine 的 `CREATE DATABASE` 只建库不建表）。把本地 `sql/` 文件通过 stdin 管道进 Pod 的客户端：

```bash
# MySQL：device + alarm 表
kubectl exec -i -n iot deploy/mysql -- mysql -uroot -pexample123 iot < sql/03_mysql_schema.sql
kubectl exec -i -n iot deploy/mysql -- mysql -uroot -pexample123 iot < sql/04_alarm_schema.sql
# TDengine：超级表 env_sensor（子表由 app 自动建）
# ⚠️ 不能用 `taos < 文件` 管道——stdin 交互模式不处理 # 注释，会把注释和 SQL 拼成非法 token。
# 用 taos -s 直接建（单条 DDL 最稳）：
kubectl exec -n iot deploy/tdengine -- taos -s "CREATE STABLE IF NOT EXISTS iot.env_sensor (ts TIMESTAMP, temp FLOAT, humi FLOAT, light INT, soil_ph FLOAT) TAGS (device_id NCHAR(32), location NCHAR(64), dev_type NCHAR(16));"
# 或把文件 cp 进去用脚本模式 taos -f（-f 才正确处理 # 注释）：
#   kubectl cp sql/01_schema.sql iot/<tdengine-pod>:/tmp/s.sql && kubectl exec -n iot <pod> -- taos -f /tmp/s.sql
```

> 更可复现的做法（重建集群零手工）：MySQL 把 schema 放 ConfigMap 挂到 `/docker-entrypoint-initdb.d/`（**仅首次初始化空数据目录时自动执行**）；TDengine 用一个 init Job 跑 `taos -f`。本地学习先手动灌，够用。
