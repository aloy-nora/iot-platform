# 03 · 问题排查

> 格式：每条按 **现象 → 原因 → 解决 → 原理(面试点)**。

---

## 1. `kubectl apply -f 目录/` 报 `namespaces "iot" not found`

- **现象**：apply 一个目录，namespace 建好了，但同批的 Secret/Deployment 等报 `namespaces "iot" not found`。
- **原因**：`apply -f 目录` 按**文件名字母序**处理。`mysql.yaml`(m) 排在 `namespace.yaml`(n) 前，先创建依赖 `iot` 的资源，而此时命名空间还没建出来。
- **解决**：
  - 立刻解围：**再 apply 一次**（声明式幂等，这次 namespace 已存在）。
  - 根治：给文件加数字前缀控制顺序 —— `00-namespace.yaml` → `10-mysql.yaml` → `20-redis.yaml`。
- **原理**：apply 目录无依赖编排；资源创建有先后依赖时，靠文件名排序或分步 apply 保证顺序。

## 2. 镜像 `ImagePullBackOff`，报 `proxyconnect tcp: dial tcp 127.0.0.1:8001: connect: connection refused`

- **现象**：Pod 卡在 `ErrImagePull` / `ImagePullBackOff`，`kubectl describe pod` 里看到上面这行。
- **原因**：kind 建集群时把宿主机 `HTTP_PROXY=127.0.0.1:8001` 继承进了节点 containerd。**但节点容器内部的 `127.0.0.1` 是节点自己，不是宿主机**，那没有代理 → 连接被拒 → 拉不到镜像。
- **解决**：不走网络，直接把宿主机已有镜像拷进节点：
  ```bash
  kind load docker-image mysql:8.0 --name iot
  docker exec iot-control-plane crictl images | grep mysql   # 确认进去了
  kubectl delete pod -n iot -l app=mysql                     # 让 Pod 重建，用本地镜像
  ```
  （镜像 tag 非 `:latest`（如 `:8.0`）默认 `imagePullPolicy: IfNotPresent`，本地有就不再拉。）
- **原理**：**容器里的 `127.0.0.1` ≠ 宿主机的 `127.0.0.1`**。这是本地代理 + kind 的经典坑。
- **深挖（2026-07-07）**：代理源头是节点里 `/etc/systemd/system.conf.d/proxy-default-environment.conf` 的**全局 `DefaultEnvironment`**（kind 从宿主机继承 `HTTP_PROXY` 写入），所有 systemd 服务（含 containerd）都继承它。
  - 外科手术式清掉：给 containerd 加 drop-in `/etc/systemd/system/containerd.service.d/no-proxy.conf` 设 `Environment="HTTP_PROXY="`（清空，per-unit 覆盖全局）→ `systemctl daemon-reload && systemctl restart containerd`。
  - 清掉后实测：**ghcr.io / quay.io 直连能拉了**；但 **Docker Hub（registry-1.docker.io）直连 TLS 握手超时**（本网络被墙）——Docker Hub 镜像仍需 `kind load`（宿主机走代理能拉，再 load 进去），或配国内 registry mirror。

## 3. 自建镜像 `ImagePullBackOff` / `ErrImageNeverPull`（后端会遇到）

- **现象**：部署本地 `docker build` 出来的镜像（Hub 上没有），Pod 拉不到。
- **原因**：**宿主机 Docker 和 kind 节点 containerd 是两套隔离的镜像库**。本地 build 的镜像只在宿主机，节点里没有、也无法从网上拉。
- **解决**：build 后先 `kind load` 送进节点，并把清单的 `imagePullPolicy` 设为 `IfNotPresent`（或 `Never`）：
  ```bash
  docker build -t iot-backend:dev .
  kind load docker-image iot-backend:dev --name iot
  kubectl apply -f deploy/k8s/20-backend.yaml
  ```
- **原理**：`docker` 打的镜像 ≠ K8s 节点能直接用；本地开发必须显式 `kind load`。

## 4. 进 kind 节点后 `docker` 命令找不到

- **现象**：`docker exec -it iot-control-plane bash` 进去后敲 `docker ps` 提示 command not found。
- **原因**：节点里的容器运行时是 **containerd**，不是 Docker（K8s 1.24 起移除 dockershim，直接对接 CRI/containerd）。
- **解决**：节点内用 `crictl`（`crictl ps` / `crictl images` / `crictl logs`）。
- **原理**：Docker 只是本地打镜像的工具；集群里跑容器是 containerd 的事。面试问「K8s 还用 Docker 吗」即答此。

## 5. 查进程真实用户：`id` 不准 / 精简镜像没 `ps`

- **现象**：想知道 `mysqld`/`redis-server` 以什么用户跑。`exec` 进去 `id` 显示 root，但不确定主进程是不是 root；mysql 容器还没有 `ps`。
- **原因**：
  - `id`/`whoami` 是**你 exec 进去那个 shell 的身份**（容器默认用户），**不等于主进程用户**。例：mysql 容器 exec 是 root，但 `mysqld` 由 entrypoint 用 gosu **降权到 mysql(uid 999)** 跑。
  - mysql 官方镜像精简，没装 `ps`。
- **解决**：
  ```bash
  # 通用（不依赖 ps）：查 1 号进程真实 uid，再映射用户名
  kubectl exec -n iot deploy/mysql -- cat /proc/1/status | grep -i '^Uid:'
  kubectl exec -n iot deploy/mysql -- getent passwd 999
  # alpine 有 ps 时直接看 USER 列
  kubectl exec -n iot deploy/redis -- ps
  ```
- **原理**：查主进程用户看进程本身（`ps` 的 USER 列 或 `/proc/1/status` 的 Uid），别信 exec 的 `id`。

## 6. redis 以 root 跑 / 降权后写不进 PVC

- **现象**：redis-server 是 root（`ps` 显示 PID 1 root，`/data` 属主 root）。若加 `runAsUser` 降权，redis 可能写不了 `/data`。
- **原因**：我们用 `command: ["sh","-c","exec redis-server ..."]` 注入密码，**绕过了镜像 entrypoint 原本的降权逻辑** → redis 以 root 跑。而 PVC 挂进来默认 root 属主，降权后无写权限。
- **解决**（做生产化版本时）：
  ```yaml
  spec:
    template:
      spec:
        securityContext:
          runAsUser: 999      # redis:7-alpine 的 redis 用户
          fsGroup: 999        # 让挂载卷属组=999，降权后才写得进 /data
  ```
  或改回不覆盖 entrypoint：密码写进 `redis.conf`，用 Secret 挂成文件，让原生 entrypoint 自动降权。
- **原理**：容器安全基线=非 root + 最小权限；卷权限靠 `fsGroup` 对齐。本地学习可暂不改。

## 7. 清单被拒：`env` 写成 compose 语法

- **现象**：`env: [- KEY=VALUE]` 部署报校验错误。
- **原因**：compose 的 env 是 `KEY=VALUE` 字符串数组；**K8s 的 env 是对象数组，必须 `name`/`value`**。从 compose 抄清单时最常踩。
- **解决**：
  ```yaml
  env:
    - name: TAOS_FQDN
      value: tdengine
  ```
- **原理**：compose 与 K8s 是两套 schema，字段结构不通用，不能照抄。

## 8. 清单被拒：卷名/资源名带下划线

- **现象**：`volumes: - name: taos_data` 报 `a lowercase RFC 1123 label must consist of lower case alphanumeric characters or '-'`。
- **原因**：K8s 里卷名、多数资源名要求是 **DNS-1123 标签**：只允许小写字母、数字、`-`，**不许下划线**。
- **解决**：`taos_data` → `taos-data`（或 `taos-storage`）。
- **原理**：K8s 命名走 DNS 规范（名字常用于 DNS/标签），下划线非法。

## 9. 一个 PVC 想挂多个目录 → 用 subPath

- **现象**：data 和 log 两个 volumeMount 都写同一个 `claimName`，结果两个路径内容相同、混在一起。
- **原因**：同一 PVC 挂到多个 mountPath，每个挂的都是**卷根**，看到的是同一份内容。
- **解决**：用 `subPath` 把一个 PVC 切成子目录：
  ```yaml
  volumeMounts:
    - { name: taos-storage, mountPath: /var/lib/taos, subPath: data }
    - { name: taos-storage, mountPath: /var/log/taos, subPath: log }
  volumes:
    - name: taos-storage
      persistentVolumeClaim: { claimName: tdengine-pvc }
  ```
- **原理**：`subPath` 指定挂卷内的子路径，实现「一卷多目录」隔离；要彻底隔离则用多个 PVC。

## 10. Service 多端口报 `spec.ports[x].name: Required value`

- **现象**：Service 有多个端口时部署报每个 port 的 `name` 必填。
- **原因**：**多端口 Service 每个 port 必须有唯一 `name`**；单端口才可省略。
- **解决**：
  ```yaml
  ports:
    - name: mqtt
      port: 1883
      targetPort: 1883
    - name: mqtt-ws
      port: 8083
      targetPort: 8083
  ```
- **原理**：多端口下 K8s 用 name 区分/引用各端口（如 `targetPort` 按名引用、监控按名聚合）。
- **对比**：**容器 `containerPort` 的 `name` 是可选的**（几个端口都不强制，整个 `ports` 块也只是信息性）；只有 **Service 的多端口**才强制命名。给容器端口起名的好处：探针 `port` / Service `targetPort` 可**按名引用**而非写死数字。

## 11. TDengine 在 K8s 上 CrashLoopBackOff（FQDN 解析）

- **现象**：tdengine Pod 反复重启；日志 `taos -h tdengine -P 6030 --check` → `0: unavailable` / `failed to SendRecv ... timeout`；探针 `taos -s` 也 1s 超时。
- **原因**：设了 `TAOS_FQDN=tdengine`，taosd 启动后要连自己 `tdengine:6030`。但 **K8s 里 `tdengine` 解析到 Service 的 ClusterIP（不是 Pod），且 Service 只暴露 6041 没有 6030** → 自检连不上 → 崩溃。（compose 能跑是因为它 `hostname: tdengine` 且名字直接解析到容器自身。）
- **解决**（单节点本地）：
  - `TAOS_FQDN=localhost` —— taosd 连自己走回环；app 仍走 Service 到 taosAdapter 的 6041（WS 驱动不需解析 taosd FQDN）。
  - 探针改 `tcpSocket: 6041` 并放宽（`initialDelaySeconds` 拉到 60、`timeoutSeconds` 加大），别用 1s 超时的重命令。
  - 改过 FQDN 后**先删 PVC 重来**：taosd 会把 FQDN 写进数据目录，残留旧值会报「FQDN changed」。`kubectl delete -f 30-tdengine.yaml` 再 apply。
- **原理**：有状态服务对「自身网络身份(FQDN)」敏感。多节点正规解法是 **StatefulSet + headless Service（`clusterIP: None`）+ `publishNotReadyAddresses`**，让名字解析到 Pod 自身 IP；单节点用 `localhost` 最省。

## 12. `taos < 文件` 灌 schema 报 `unrecognized token: # ...`

- **现象**：`kubectl exec -i deploy/tdengine -- taos < 01_schema.sql`，报把 `# 注释` 和后面的 SQL 拼成一条的 `unrecognized token`，超级表没建成。
- **原因**：**taos 从 stdin 进来是交互模式，不按脚本模式处理 `#` 注释**，注释行和下一条 SQL 被拼到一起 → 非法。
- **解决**：
  - 单条 DDL 用 `taos -s "CREATE STABLE ..."`（最稳，无注释无文件）；
  - 或 `kubectl cp` 文件进容器后用 `taos -f 文件`（`-f` 脚本模式才正确处理 `#` 注释）。
- **原理**：mysql 客户端管道能吃注释，taos stdin 不行——不同客户端对管道/注释的处理不一样，灌 schema 优先用各自的「脚本文件模式」。
