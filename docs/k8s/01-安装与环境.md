# 01 · 安装与环境

## 工具清单

| 工具 | 作用 | 装法 |
|---|---|---|
| Docker Desktop | 跑容器、给 kind 当底座 | 官网下载 |
| kubectl | 操作 K8s 集群 | `brew install kubectl` |
| kind | 用 Docker 起本地 K8s 集群 | `brew install kind` |
| helm | K8s 包管理（后续打包用） | `brew install helm` |

## 从零到集群

```bash
# 0. 先启动 Docker Desktop（GUI），等小鲸鱼图标稳定
open -a Docker

# 1. 装工具
brew install kind helm

# 2. 建集群（名字叫 iot）
kind create cluster --name iot

# 3. 验证
kubectl get nodes -o wide          # 节点 Ready
kubectl config current-context     # 应为 kind-iot
kind get clusters                  # 列出 iot
```

## ⚠️ 环境坑：本地代理导致节点拉不到镜像

本机配了 `http_proxy=http://127.0.0.1:8001`。**kind 建集群时会把宿主机的 `HTTP_PROXY` 继承进节点的 containerd**，
但 `127.0.0.1` 在节点容器内部指的是「节点自己」而非宿主机 → 代理不可达 → 所有镜像 `ImagePullBackOff`。

**应对（本项目采用）**：不让节点走网络拉镜像，改用 `kind load` 把宿主机已有的镜像拷进节点。详见 `03-问题排查.md`。

> 根治方案（暂未做）：重建集群时正确配置代理（指向 `host.docker.internal` 或宿主机网关），或配国内镜像加速。
> 等做「生产化」版本时再处理。

## 三层套娃心智模型

```
宿主机 Docker
  └── iot-control-plane   ← kind 节点（本身是宿主机上的一个 docker 容器）
        └── containerd    ← 节点内部的容器运行时（不是 docker！）
              └── mysql / redis 等 Pod 容器
```

- 宿主机层：`docker ps` 看得到节点容器 `iot-control-plane`。
- 节点内部：运行时是 **containerd**，**没有 docker 命令**，查东西用 `crictl`（见 `02`）。
