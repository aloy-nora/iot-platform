# 06 · 概念与原理

底层概念/原理,支撑其他文档里的操作。面试常考。

## PID 1、`exec`、tini —— 容器优雅停机

### PID 1 的两个特殊行为
1. **信号默认动作被内核屏蔽**:内核不对 PID 1 施加信号默认处理。PID 1 若**自己没注册 SIGTERM 处理**,收到 SIGTERM 会当没看见 → 不退出 → 宽限期后被 SIGKILL。
2. **僵尸回收**:PID 1 有义务收养并 reap 孤儿进程,否则僵尸堆积。

### 什么进程能安心当 PID 1
**自己处理信号 + 自己 reap 子进程的应用**(redis / nginx / mysql 等)→ 当 PID 1 没问题,这本就是它们跑官方镜像时的常态。
- 例:redis 自己注册了 SIGTERM(优雅落盘退出)、自己 wait 掉 RDB/AOF 子进程。

### `exec` 防的是"裸 sh 当 PID 1"
| 写法 | PID 1 | 结果 |
|---|---|---|
| `sh -c "redis-server ..."`(无 exec) | sh | sh 不转发信号 + PID1 屏蔽默认动作 → SIGTERM 被吞,redis 收不到 → 硬等超时被 KILL ❌ |
| `sh -c "exec redis-server ..."` | redis | redis 顶替 sh,自处理 SIGTERM → 优雅退出 ✅ |
| `tini -- redis-server ...` | tini | tini 转发信号给 redis + reap 僵尸 → 也 ✅ |

### 什么时候才需要 tini
主进程**自己不处理信号**(收到 SIGTERM 干瞪眼)或**会留孤儿/僵尸**(fork 一堆子进程又不 reap)时,用 tini 当 PID 1 帮它转发信号 + 收尸。
- **一句话**:能自理的应用 → `exec` 让它当 PID 1;不能自理的 → tini 兜底。redis 属前者,不用 tini。

## `kubectl exec` 的 `--`

`--` 是 POSIX 惯例:"**选项到此为止,后面全是容器内要跑的命令**"。

```bash
kubectl exec -it -n iot deploy/redis -- redis-cli ping
#          └── kubectl flags ──┘       └── 容器内命令 ──┘
```

- **命令带自己的 `-` 参数时必须加 `--`**(如 `redis-cli -a ... ping`、`ls -l`、`mysql -e ...`),否则 kubectl 会把这些参数当成自己的 → 报错。
- 纯裸命令(`kubectl exec pod ls`)技术上能跑,但**不带 `--` 的写法已 DEPRECATED**,会打印警告。
- 记法:`-it` 等开关放 `--` 前,容器内命令放 `--` 后。**养成习惯永远加 `--`。**

## Dockerfile 的 ENTRYPOINT vs CMD

- **ENTRYPOINT**:固定的入口程序,一般不被覆盖。
- **CMD**:默认参数(或默认命令),`docker run` 时可轻易覆盖。
- 经典搭配:`ENTRYPOINT ["redis-server"]` + `CMD ["--port","6379"]` → 跑 `redis-server --port 6379`;`docker run img --port 7000` 只替掉 CMD。
- `docker run img <东西>` 覆盖的是 **CMD**(追加在 ENTRYPOINT 后);换 ENTRYPOINT 要用 `docker run --entrypoint`。
- 两种写法:
  - exec 形式 `ENTRYPOINT ["redis-server"]` → 直接 exec,程序是 PID 1,能收信号 ✅
  - shell 形式 `ENTRYPOINT redis-server` → 实际是 `/bin/sh -c redis-server`,程序成 sh 子进程 → 信号坑 ❌

## K8s command/args ↔ Docker ENTRYPOINT/CMD（易错!）

**Dockerfile 里没有 "command"/"args" 这两个词**,只有 ENTRYPOINT 和 CMD。K8s 用另一套名字覆盖它们:

| 概念 | Docker | K8s |
|---|---|---|
| 入口程序 | **ENTRYPOINT** | **command** |
| 默认参数 | **CMD** | **args** |

- K8s `command` 覆盖镜像 `ENTRYPOINT`;K8s `args` 覆盖镜像 `CMD`。
- **不是**"新 command 盖老 command"——两边名字不同,这是最常见的误解。

覆盖规则:

| 镜像 ENTRYPOINT | 镜像 CMD | k8s command | k8s args | 实际运行 |
|---|---|---|---|---|
| `[ep]` | `[cmd]` | 不写 | 不写 | `ep cmd` |
| `[ep]` | `[cmd]` | `[c]` | 不写 | `c`（镜像 CMD 被丢弃） |
| `[ep]` | `[cmd]` | 不写 | `[a]` | `ep a` |
| `[ep]` | `[cmd]` | `[c]` | `[a]` | `c a` |

> 应用:redis 清单写 `command: ["sh","-c",...]` **覆盖了镜像 ENTRYPOINT(docker-entrypoint.sh)**,所以它原本的降权逻辑没执行 → redis 以 root 跑(见 `03` 第 6 条)。
